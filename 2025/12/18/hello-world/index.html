4，xctf_ret2libc ——32

*很疑惑的是，本地拿不到shell但在靶机居然可以？

*通过ldd 命令查看题的libc版本（一般其实会给libc版本）

xctf_ret2libc 中，本地拿不到 shell 但远程可以的核心原因是 环境差异（尤其是 libc 版本匹配度），但漏洞利用的底层逻辑不受影响 —— 由于系统开启了 ASLR（地址空间随机化），libc 库每次加载到内存的基地址都会随机变化，导致 system、/bin/sh 等关键符号的绝对地址无法提前预知。因此，必须通过以下步骤推导真实地址：

  1. 利用漏洞（如栈溢出）泄露 libc 中某个 已知符号的真实地址（通常选择 write、read 等程序已调用过的函数）—— 这些函数的 GOT 表地址（write_got）是程序编译时固定的（非随机），且程序运行后会被填充为 libc 中该函数的真实内存地址。
  2. 用泄露的 真实地址 - 该符号在本地 libc 中的偏移量，计算出远程靶机上 libc 的加载基地址（libc_base = 泄露的真实地址 - 本地libc.symbols['目标符号']）。
  3. 基于 libc_base，分别计算 system 函数和 /bin/sh 字符串的真实地址（system_addr = libc_base + 本地libc.symbols['system']，bin_sh_addr = libc_base + 本地libc中/bin/sh的偏移）。
  4. 再次构造栈溢出 payload，将返回地址覆盖为 system_addr，并传入 bin_sh_addr 作为参数，最终执行 system("/bin/sh") 拿到 shell
  5. 总之 随机址化貌似像个整体平移地址 ？

18，useUAF

具体来说：

  1. add(32, ...) 两次：创建两个同 size 的 chunk，满足 fastbin「同 size 才会进同一链表」的规则；
  2. d(0)：chunk0 被释放，进入 32 字节 fastbin 链表 → 链表：chunk0→NULL；
  3. d(1)：chunk1 被释放，进入 32 字节 fastbin 链表 → 链表：chunk1→chunk0→NULL；

这两步释放后，32 字节 fastbin 链表里有两个空闲 chunk，且：

  • chunk1 的 fd = chunk0（堆管理器自动设置）；
  • chunk0 的 fd = NULL；
  • 链表是线性的（无循环），但这对你的利用已经足够。

第二步：为什么要这么做？（你的 wp 后续利用的关键）

你接下来执行 add('8',p32(backdoor))—— 申请 8 字节 chunk，但 8 字节 fastbin 链表原本是空的，堆管理器会做什么？

核心知识点：fastbin 的「size 幻视」（你的 wp 利用的核心）

glibc 堆管理器对 fastbin 的 size 检查不严格：如果小 size（8 字节）的 fastbin 链表为空，堆管理器可能会 “错误复用” 大 size（32 字节）链表中的 chunk（取决于 glibc 版本 / 编译选项）。

你的 wp 正是利用这一点：

  1. 32 字节 fastbin 链表有两个空闲 chunk（chunk1→chunk0）；
  2. 你申请 8 字节 chunk，堆管理器 “误拿” 32 字节链表的头部 chunk1；
  3. 你发送的 p32(backdoor) 会写入 chunk1 的内存，覆盖它的 fd 指针；
  4. 因为 chunk1 的 fd 原本指向 chunk0，现在被改成 backdoor 地址 → 当你调用 u(0) 时，程序读取 chunk0 关联的 fd（实际是 chunk1 被篡改的 fd），执行 backdoor。

第三步：总结你的 wp 里这几步的意义（一句话讲透）

你写 add两个32字节chunk + d(0)→d(1)，是为了在 32 字节 fastbin 链表中制造两个空闲 chunk，让后续申请 8 字节 chunk 时，堆管理器能 “误复用” 这个大 size 链表的 chunk，从而让你有机会覆盖 fd 为 backdoor 地址，最终通过 u(0) 触发调用。

19，polar_Emo_chunk

步骤 1：利用堆溢出制造堆块重叠（Chunk Overlapping）

通过 edit 功能的堆溢出，覆盖下一个 chunk 的size字段：

  1. 申请多个相邻 chunk，编辑前一个 chunk 时故意溢出，将后一个 chunk 的 size 改为small chunk 范围（超出 fastbin），并篡改 size 字段的标志位（如 PREV_INUSE）；
  2. 释放被篡改 size 的 chunk，它会被放入unsortedbin（而非 fastbin），而 unsortedbin 中的 chunk 的 fd/bk 指针会指向 libc 的main_arena+88位置，为后续泄露做铺垫。

步骤 2：泄露 libc 基址（unsortedbin 信息泄露）

利用程序的 print 打印功能，读取 unsortedbin 中 chunk 的 fd/bk 指针：

  1. 从 unsortedbin 中切割申请一个小 chunk，该 chunk 不会清空原有指向main_arena+88的指针；
  2. 调用 print 功能打印该 chunk 内容，得到main_arena+88的真实地址；
  3. 用该地址减去本地 glibc 中 main_arena+88 的固定偏移，计算出libc 基址，进而得到__malloc_hook/__realloc_hook/one_gadget 的真实地址。

步骤 3：fastbin 攻击，精准布局到 malloc_hook 附近

利用 fastbin 的分配特性，向__malloc_hook-0x23位置申请 fake chunk（核心绕开 fastbin 的 size 检查）：

  1. 将 unsortedbin 中剩余的 chunk 释放，让其进入 fastbin；
  2. 再次利用堆溢出，修改该 fastbin 中 chunk 的fd 指针，指向__malloc_hook-0x23（该地址的 size 字段天然满足 fastbin 的 0x70 链表检查，可绕过 fastbin 的 size 验证）；
  3. 连续申请 chunk，fastbin 会根据 fd 指针的指向，直接将__malloc_hook-0x23位置分配为新 chunk，实现任意地址写。

步骤 4：覆写钩子函数，触发恶意执行（核心用 realloc_hook 调栈）

这一步是视频的重点（也是该题的关键技巧，区别于普通堆题），不用直接覆写 malloc_hook，而是通过「malloc_hook -> realloc_hook -> one_gadget」的二级跳 实现执行：

  1. 将__realloc_hook覆写为one_gadget 地址（无对齐限制，成功率 100%）；
  2. 将__malloc_hook覆写为**__realloc_hook 的地址**；
  3. 触发程序调用malloc()（如 add 操作），此时会先执行__malloc_hook指向的__realloc_hook，再跳转到 one_gadget，直接 getshell。

25，easyheap

这题与上述的找malloc_hook方法不同，区别就在于这个题他没有打印的功能，所以我们无法泄露arena_main+0x88的地址，就无法拿到libc基址。

这题的思路主要分为4个步骤：

1，通过堆溢出修改fd地址，跳到heaparry[0] 拿下chunk0的控制权

2，通过malloc，修改其chunk0为free@got地址 这样只需要修改chunk0就可以控制free@got

3，通过步骤2我们控制了free@got ，写入system@plt地址构造shellcode

4，调用free函数 free@plt ->free@got ->system@plt ->system@got 就是调用了system 此时free(chunk[0]) ==> system(chunk[0]) 由于我们提前布置好了”/bin/sh\x00”，得到了shell
